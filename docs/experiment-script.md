# Experiment Script

## 1. Opening Script
Thank you for helping us test our prototype. This is not a test of you. We are testing whether our product idea and interface are clear. Please think aloud while using the prototype.

## 2. Test Scenario
You are an RSU student who needs academic equipment, a learning resource, or a student service for a university project. You want to use RSU Nexus to search for a suitable listing, view its details, and submit a request to the provider.

## 3. Tasks for Tester

| Task No. | Task Instruction | Related Requirement | Expected Result |
|---|---|---|---|
| T1 | Open the homepage and explain what the system is for. | FR-01, FR-02 | User understands that RSU Nexus is a campus resource marketplace for RSU students. |
| T2 | Browse the marketplace and find a suitable resource or student service. | FR-05, FR-06 | User can browse, search, and filter marketplace listings. |
| T3 | Open one listing and review its information. | FR-07, FR-08 | User understands the listing details, provider information, offer terms, and availability status. |
| T4 | Contact the provider or submit a request for the selected listing. | FR-03, FR-10, FR-11 | User completes the request form and receives a confirmation message. |
| T5 | Check the submitted request and identify its current status. | FR-08 | User understands whether the request is Pending, Approved, Rejected, Completed, or Closed. |
| T6 | Create a new equipment, learning-resource, or student-service listing. | FR-03, FR-10, FR-11 | User completes the listing form, understands required fields, and sees a successful submission message. |

## 4. Observation Notes
Record where the tester hesitates, clicks the wrong place, asks questions, misunderstands a label, or gives useful feedback.

Also observe whether the tester can:

- Understand the purpose of RSU Nexus without assistance.
- Find the marketplace search and filter controls.
- Understand the difference between Equipment, Learning Resources, and Student Services.
- Recognize the listing availability and request-status labels.
- Identify the correct action for contacting a provider or submitting a request.
- Complete forms without missing required information.

## 5. Closing Questions

- What was easy to understand?
- What was confusing?
- Which feature was most useful?
- Was it easy to find the resource or service you needed?
- Were the availability and request-status labels clear?
- Did you understand how to contact the provider or submit a request?
- What should be improved before the final prototype?
- Would you use RSU Nexus as an RSU student? Why or why not?
